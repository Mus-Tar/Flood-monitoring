## 1. 项目现状速览（已完成能力）

当前项目是一个前后端分离的洪水监测系统，技术栈明确，功能链路已经具备“可演示基础”：

- 前端：`Vue3 + Vite + Tailwind + ECharts + Leaflet`
- 后端：`Spring Boot 2.7 + MyBatis-Plus + MySQL`
- 现有模块：
  - 用户注册/登录/角色区分
  - 监测点管理（增删改查）
  - 数据导入（CSV/XLSX）

  - 预警事件处理（确认/解除/删除）
  - 历史数据图表分析

结论：**项目不空、功能不少，已经超过“课程作业级”雏形，具备继续打磨成比赛作品的基础。**

## 2. 关键短板（会被评委追问/扣分的点）

### 2.1 安全与权限是“演示级”而非“上线级”

- 登录返回固定 `demo-token`，非真实鉴权链路。
- 密码明文存储/比对。
- 权限主要在前端路由守卫，后端接口缺少强制鉴权拦截。
- CORS 过宽（`*` + credentials）。

风险：评委一问“如何防越权/防伪造请求”，当前答案会比较弱。

### 2.2 “预测”模块技术说服力不够

- 后端存在 `LstmPredictorStub` / `RfPredictorStub`，但属于占位逻辑。
- 预测页当前核心是“把已有时间段数据拆分成历史/未来展示”，不是严格意义的模型推理。

风险：评委会问“模型如何训练、如何评估、误差多少”，当前难支撑。

### 2.3 缺少真正实时链路

- 当前主要是手工导入数据。
- 没有定时采集任务、没有消息流、没有 WebSocket/SSE 推送。

风险：系统名义是“监测预警”，但实时性证据不足。

### 2.4 工程化不足

- 缺少自动化测试（单测/接口测试）。
- 缺少环境分离与统一部署文档（dev/test/prod）。
- 前端 API 地址写死 `localhost`，上服务器需要手动改。
- 文本存在编码异常（界面中文乱码风险），会影响演示观感。

### 2.5 数据治理能力薄弱

- 采集数据缺少去重与质量校验闭环（如同站点同时间重复值、突变异常值）。
- 监测数据虽然有索引，但缺少更完整的数据生命周期管理（冷热分层、归档策略）。

## 3. 参赛优化方向（按优先级分层）

## P0（必须先做，决定“能不能打比赛”）

1. 鉴权闭环升级（1周内）
- 引入 JWT（登录签发、接口验签、过期机制、刷新机制）。
- 密码改为 BCrypt。
- 后端按角色控制接口（ADMIN/USER），前端权限只做展示层。

2. 预测模块“去伪存真”（1-2周）
- 明确两种模式：
  - `展示模式`（历史趋势回放）
  - `模型模式`（真实预测）
- 至少落地一个可解释基线模型（如 ARIMA/XGBoost/LSTM 任选其一）。
- 输出指标：MAE/RMSE/MAPE，并在页面可视化。

3. 实时采集最小闭环（1周）
- 增加 `scheduler` 定时任务（5/10分钟）拉取外部水情。
- 数据入库后自动触发阈值评估。
- 前端增加“最近更新时间、数据源状态、采集成功率”。

4. 演示稳定性修复（3-5天）
- 统一 UTF-8 编码，清理乱码文本。
- 前端改为环境变量配置 API 地址。
- 增加全局错误提示与兜底页面。

## P1（拉开差距，决定“能不能拿奖”）

1. 实时推送能力
- 后端加 WebSocket/SSE，预警事件秒级推送到前端。
- 大屏新增“最新告警滚动流”。

2. 数据质量与可信度
- 增加异常值检测（突跳/缺失/重复）。
- 数据质量评分（完整率、连续率、可信度）并可视化。

3. 预警策略增强
- 从“单指标阈值”升级到“多因子规则”（水位+雨量+持续时长）。
- 加入“动态阈值/季节阈值”策略对比。

4. 比赛叙事增强
- 新增“事件复盘”页面：某次强降雨期间预警全过程回放。
- 增加“策略仿真”按钮：调整阈值后对历史事件回放对比。

## P2（冲击高分，决定“创新亮点”）

1. 模型工程化
- 模型服务化（Python/FastAPI 或 Java 推理服务）。
- 模型版本管理、在线切换与 A/B 对比。

2. 数字孪生表达
- 在 GIS 上叠加流域分层、风险热力、未来趋势动画。

3. 多源异构融合
- 国家流域 + 省级站点 + 自建传感器三源融合。

## 4. 服务器部署建议（可落地）

推荐最小生产架构：

- `Nginx`：静态资源 + 反向代理
- `Spring Boot`：业务 API
- `MySQL`：核心数据
- `Redis`（建议）：缓存热点站点、限流、会话/令牌黑名单
- `Scheduler/Adapter`：外部水情数据采集服务

部署方式建议：

1. Docker Compose（比赛/答辩最快）
- `nginx + backend + mysql + redis` 一键拉起。
- 可快速在评委机/云服务器复现。

2. 生产化（后续）
- 前后端分离部署，按环境配置（dev/test/prod）。
- 加日志链路（请求日志、任务日志、告警日志）与健康检查。

必须补齐的部署项：

- `application-prod.yml`
- `.env` / 前端 `.env.production`
- 数据库初始化与迁移脚本（建议 Flyway）
- 一份可执行 `README-DEPLOY.md`

## 5. 接入“国家大江大河实时水情”的路线图

先讲现实：**国家级/流域级实时水情数据，很多并不是完全公开标准 API，常见情况是授权接口 + 门户展示 + 多源异构。**

建议采用“三层接入架构”：

1. Source Layer（数据源层）
- 官方可授权接口（优先）
- 公开门户数据（作为补充）
- 自有监测设备/人工导入（兜底）

2. Adapter Layer（适配层）
- 每个来源写独立 Adapter：
  - 拉取
  - 字段映射
  - 去重（`source + station_code + timestamp`）
  - 质量校验
- 统一落地到标准表 `external_observation_raw` + `monitor_data`。

3. Service Layer（业务层）
- 统一对外 API（前端只调用你自己的接口，不直接绑外部源）。
- 统一预警评估逻辑与推送逻辑。

建议的数据标准字段：

- `source`（来源）
- `basin`（流域）
- `station_code` / `station_name`
- `obs_time`
- `water_level` / `rainfall` / `flow`
- `quality_flag`（质量标记）
- `ingest_time`

实时策略建议：

- 采集频率：5~10 分钟
- 告警判定：入库后立即触发
- 前端刷新：WebSocket/SSE 实时推送，轮询作兜底
